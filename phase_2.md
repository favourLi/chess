# 第二阶段功能规格说明（在线系统）

> **文档版本**: 1.0  
> **对应计划**: `PROJECT_PLAN.md` — 第二阶段：在线系统（2–3 个月）  
> **前置条件**: 第一阶段「核心基础」已完成（本地双人对战、规则引擎、基础 3D 与 UI）

---

## 1. 文档目的与范围

### 1.1 目的

在第一阶段单机/同机双人能力之上，定义第二阶段需交付的**功能边界、业务流程、数据与接口约定、验收标准**，供产品、前后端与测试对齐。

### 1.2 范围（In Scope）


| 模块       | 说明                     |
| -------- | ---------------------- |
| 用户系统     | 注册、登录、资料、等级、JWT 会话     |
| 在线匹配     | 快速匹配、房间、好友、好友对战        |
| 实时对战     | WebSocket、状态同步、断线重连、观战 |
| 主题系统     | 棋盘/棋子主题、切换、自定义颜色       |
| 回放系统     | 对局存档、步骤回放、播放控制         |
| 3D UI 完善 | 交互与动效、体验优化（与在线流程衔接）    |


### 1.3 非目标（Out of Scope，属第三阶段或后续）

- 大规模性能压测与全机型兼容矩阵（第三阶段）  
- 反作弊深度策略、完整运维监控（第三阶段）  
- 付费、广告、排位赛季等商业化能力

## 2. 与第一阶段的衔接

### 2.1 复用能力

- **规则引擎**：走子、吃子、将军/将死、回合判定仍以服务端为最终权威（在线模式下）；客户端仅做预判与表现。  
- **3D 场景**：棋盘、棋子、相机动画、皮肤预设可继续复用；在线需增加「同步状态 → 驱动本地表现」的一层。  
- **本地 UI 模式**：保留「本地双人」入口，与「在线大厅 / 房间」并列，避免强制联网。

### 2.2 架构原则

- **权威在服务端**：棋局 FEN/着法序列、胜负、超时等由服务器校验与广播。  
- **弱网友好**：关键操作可重试、状态可恢复；断线重连后拉取完整局面。  
- **可观测**：关键事件（登录、匹配、开局、走子、终局）具备日志与关联 ID（如 `gameId`、`userId`）。

## 3. 用户系统

### 3.1 目标

为在线匹配、好友、回放归属提供稳定身份；会话采用 **JWT**（或 Access + Refresh 组合，下文统称「令牌体系」）。

### 3.2 功能列表

#### 3.2.1 注册


| 项   | 说明                            |
| --- | ----------------------------- |
| 输入  | 用户名（或邮箱/手机号，按产品定一种主键）、密码、可选昵称 |
| 校验  | 用户名唯一、密码强度策略、防机器人（可选：验证码）     |
| 输出  | 注册成功 → 可选自动登录或跳转登录页           |
| 错误  | 重复账号、格式非法、服务不可用               |


#### 3.2.2 登录


| 项   | 说明                                                                            |
| --- | ----------------------------------------------------------------------------- |
| 输入  | 账号、密码                                                                         |
| 输出  | 颁发 **Access Token**（短期）+ 建议 **Refresh Token**（长期，httpOnly Cookie 或安全存储策略需文档化） |
| 安全  | 密码哈希存储（如 bcrypt/argon2）；登录失败次数限制；HTTPS                                        |


#### 3.2.3 个人资料管理


| 项   | 说明                         |
| --- | -------------------------- |
| 展示  | 昵称、头像 URL、用户 ID、等级/经验（若启用） |
| 编辑  | 昵称、头像；敏感项（绑定手机/邮箱）可作为迭代项   |
| 权限  | 仅本人可改；他人仅可见公开字段            |


#### 3.2.4 用户等级系统


| 项      | 说明                               |
| ------ | -------------------------------- |
| 目的     | 增强留存与成就感；可与回放、对局数挂钩              |
| 规则（示例） | 完成 N 局、胜利 M 局、登录天数等获得经验；等级段位表可配置 |
| 展示     | 个人资料、房间列表、对局结束结算页                |


#### 3.2.5 JWT 认证


| 项            | 说明                                                                                      |
| ------------ | --------------------------------------------------------------------------------------- |
| Access Token | 附在 `Authorization: Bearer` 或约定 Cookie；WebSocket 连接时携带（查询参数首包 / 子协议 / 首条 auth 消息三选一，需统一） |
| 刷新           | Access 过期后用 Refresh 换新；失败则要求重新登录                                                        |
| 吊销           | 修改密码、封号时令 Refresh 失效（服务端黑名单或版本号）                                                        |


### 3.3 验收标准（用户系统）

- 注册/登录/登出流程完整，错误提示明确。  
- 受保护 REST 接口无有效令牌时返回 **401**；令牌过期可刷新或引导登录。  
- 资料修改后多端（同账号）拉取一致。  
- 等级数值与对局结果联动逻辑可配置或可测。

## 4. 在线匹配

### 4.1 目标

让玩家以**低摩擦**进入在线对局，并支持**房间号/邀请**与**好友**两种社交路径。

### 4.2 快速匹配


| 项   | 说明                                   |
| --- | ------------------------------------ |
| 入口  | 大厅「快速开始」                             |
| 流程  | 进入匹配队列 → 匹配到对手 → 双方确认（可选）→ 创建对局并进入棋盘 |
| 规则  | 按等级段/隐藏分（可选）、先后手随机或偏好                |
| 取消  | 匹配中可取消；超时未匹配到提示重试                    |


### 4.3 自定义房间


| 项   | 说明                       |
| --- | ------------------------ |
| 创建  | 生成 **房间号**（6–8 位易读）或分享链接 |
| 加入  | 输入房间号加入；满员（2 人）后房主可「开始」  |
| 角色  | 房主：踢人（可选）、开始、解散；访客：准备/离开 |
| 状态  | 等待中 / 对局中 / 已结束          |


### 4.4 好友列表


| 项     | 说明                      |
| ----- | ----------------------- |
| 添加    | 按用户 ID / 昵称搜索；发起申请；对方同意 |
| 列表    | 在线状态、最近一局时间（可选）         |
| 删除/拉黑 | 删除好友；拉黑后不接收邀请（策略可配）     |


### 4.5 好友对战


| 项       | 说明                         |
| ------- | -------------------------- |
| 发起      | 从好友列表「邀请对战」→ 对方接受 → 进入同一房间 |
| 与快速匹配差异 | 无需全局队列，直连房间逻辑即可复用          |


### 4.6 验收标准（在线匹配）

- 两客户端可通过快速匹配进入同一 `gameId` 并正常走子。  
- 房间号加入与房主开始流程无死锁（一人离开可恢复）。  
- 好友申请/同意/拒收状态正确同步。

## 5. 实时对战

### 5.1 目标

基于 **WebSocket**（如 Socket.IO）实现低延迟、可恢复的棋局同步，并支持**观战**。

### 5.2 WebSocket 连接


| 项    | 说明                            |
| ---- | ----------------------------- |
| 连接时机 | 进入大厅即可长连，或进入房间/对局时连（需统一策略）    |
| 鉴权   | 首包或 HTTP 升级前完成身份校验            |
| 心跳   | 客户端 ping / 服务端 pong；超时断开并触发重连 |


### 5.3 棋局状态同步


| 消息类型（示例）            | 方向  | 说明                     |
| ------------------- | --- | ---------------------- |
| `game:state`        | S→C | 完整局面（FEN + 回合 + 计时器等）  |
| `game:move`         | C→S | 着法（起止坐标或 SAN）；服务端校验后广播 |
| `game:move_result`  | S→C | 成功/失败原因                |
| `game:end`          | S→C | 胜负、和棋、认输、超时            |
| `game:sync_request` | C→S | 客户端请求全量状态（重连后）         |


**原则**：客户端发送意图，服务端校验后广播权威结果；禁止仅信任客户端终局状态。

### 5.4 断线重连


| 项   | 说明                              |
| --- | ------------------------------- |
| 触发  | 网络闪断、切后台、刷新页面                   |
| 行为  | 自动或手动重连；重连成功后拉取 `game:state`    |
| 超时  | 一方长时间离线可判负或允许「托管/暂停」（产品二选一，需写明） |
| 观战者 | 重连后同样拉取最新状态                     |


### 5.5 观战模式


| 项   | 说明                     |
| --- | ---------------------- |
| 进入  | 房间允许观战 / 分享观战链接；观战者只读  |
| 能力  | 看棋盘同步、看步数列表；不可走子       |
| 人数  | 上限可配置（如 50），防止滥用       |
| 延迟  | 可与对局端同播或轻微延迟（防窥屏作弊，可选） |


### 5.6 验收标准（实时对战）

- 双端走子、吃子、将军提示与第一阶段规则一致且服务端一致。  
- 任意一端刷新后 10 秒内可恢复同一局且局面一致。  
- 观战端看到的步序与对局端一致。  
- 非法着法被服务端拒绝且对手端不受影响。

## 6. 主题系统

### 6.1 目标

在第一阶段已有皮肤/材质能力上，将**主题**产品化：可选预设 + 用户级偏好持久化 + 自定义颜色。

### 6.2 多种棋盘主题


| 项   | 说明                               |
| --- | -------------------------------- |
| 内容  | 木纹、简约、奇幻、战争等（与现有风格 key 对齐或扩展）    |
| 表现  | 棋盘 mesh 材质、可选边框/河界样式             |
| 存储  | 用户默认主题写入资料或独立 `user_preferences` |


### 6.3 多种棋子主题


| 项   | 说明                                                            |
| --- | ------------------------------------------------------------- |
| 内容  | 与棋盘主题成套或独立组合（如「棋盘 A + 棋子 B」）                                  |
| 约束  | 在线对局双方**各自本地渲染**对方所选主题或统一房主主题（产品二选一，建议：**各显示己方偏好，棋盘同步用逻辑坐标**） |


### 6.4 主题切换功能


| 项   | 说明                            |
| --- | ----------------------------- |
| 入口  | 设置页 / 大厅侧边栏；对局中是否允许切换（建议：仅局外） |
| 生效  | 立即刷新材质与环境贴图；无需重启游戏            |


### 6.5 自定义颜色


| 项   | 说明                             |
| --- | ------------------------------ |
| 范围  | 例如棋盘底色、强调色、棋子描边（在不影响棋面可读性的前提下） |
| 持久化 | 与账号绑定；未登录则仅存本地                 |
| 校验  | 对比度下限，避免红绿不分                   |


### 6.6 验收标准（主题系统）

- 至少 N 套（建议 ≥4）预设可切换且无资源加载错误。  
- 登录后换设备可拉取默认主题（若服务端已存）。  
- 自定义颜色保存后再进入对局保持一致。

## 7. 回放系统

### 7.1 目标

将对局转为**可序列化记录**，支持步骤回放与播放控制，用于复盘与学习；为第三阶段「分享、统计」打基础。

### 7.2 对局记录保存


| 字段（示例）                  | 说明                                |
| ----------------------- | --------------------------------- |
| `gameId`                | 唯一对局 ID                           |
| `players`               | 双方 userId、先后手、结果                  |
| `startedAt` / `endedAt` | 时间戳                               |
| `moves[]`               | 有序着法列表（建议 SAN 或 `from-to` + 晋升标记） |
| `initialFEN`            | 起始局面（默认标准开局）                      |
| `meta`                  | 房间类型、是否计时、版本号                     |


**存储**：服务端为主；客户端可缓存最近 K 局本地草稿（断网时暂存，可选）。

### 7.3 步骤回放


| 项     | 说明                                |
| ----- | --------------------------------- |
| 展示    | 复用 3D 棋盘，按 `moves` 逐步应用           |
| 同步    | 高亮上一手、可选箭头/坐标注释（迭代）               |
| 与在线关系 | 终局或进行中保存快照的规则需定义（仅终局保存 vs 支持中途保存） |


### 7.4 回放控制


| 项    | 说明                  |
| ---- | ------------------- |
| 控制条  | 首步、上一步、播放/暂停、下一步、末步 |
| 播放速度 | 0.5x / 1x / 2x      |
| 跳转   | 步数滑块或列表点选           |


### 7.5 权限与列表


| 项    | 说明                        |
| ---- | ------------------------- |
| 我的对局 | 登录用户可见自己参与的对局列表           |
| 他人对局 | 默认仅本人；若开放「公开回放」需单独字段与 URL |


### 7.6 验收标准（回放系统）

- 任意一局保存后，回放 100% 复现终局局面与结果。  
- 控制条在边界（首/末步）行为正确。  
- 列表分页或懒加载不阻塞主流程。

## 8. 3D UI 完善（第二阶段范围）

### 8.1 目标

支撑在线流程的**状态可见性**与**反馈**，在不大改架构前提下提升完成度。

### 8.2 功能要点


| 项        | 说明                      |
| -------- | ----------------------- |
| 大厅与房间 UI | 匹配中动画、房间成员卡片、准备状态       |
| 网络状态     | 连接中 / 已连接 / 重连中 / 失败可重试 |
| 对局内提示    | 轮到谁、对方思考中、断线提示、观战标识     |
| 动效       | 匹配成功、进入棋盘、终局结算过渡        |
| 无障碍与可读性  | 关键按钮有文字标签；色弱模式可作为迭代     |


### 8.3 验收标准（3D UI）

- 在线全流程（登录→匹配→对局→结束）无需查看控制台即可完成。  
- 断线、匹配超时、非法着法均有用户可见反馈。

## 9. 非功能需求（第二阶段基线）


| 类别  | 指标/要求                                                      |
| --- | ---------------------------------------------------------- |
| 延迟  | 同城典型网络下，着法广播 RTT 感知 < 200ms（P95 目标，可按环境调整）                 |
| 可用性 | 核心接口与 WS 服务具备健康检查；计划内维护窗口                                  |
| 安全  | HTTPS；JWT 密钥轮换策略；房间号防枚举（速率限制）                              |
| 兼容  | 与 `PROJECT_PLAN.md` 第三阶段全量兼容测试前，至少覆盖 Chrome + Edge 最新两个大版本 |
| 数据  | 用户密码与令牌不可明文落库；日志脱敏                                         |
